# Results guide

This guide explains which artifacts answer which research question, how a run directory is organized, and how to interpret the reported metrics without reversing their direction.

## Start with the result, then follow the evidence

| Question | First artifact | Deeper evidence |
|---|---|---|
| How large is the zero-shot family gap? | `results/zero_shot/zero_shot_summary.csv` | Per-structure `results/zero_shot/{oxide,nitride}/predictions.csv` |
| Did partial fine-tuning beat zero-shot? | `results/summaries/a1_protocol_1_effect_sizes.csv` | Protocol summaries and per-run test predictions |
| Was pretrained initialization useful? | Figure 3 and Protocol 1 scratch summaries | Matched N = 50 and 500 run directories |
| Were conclusions robust to optimization settings? | `results/summaries/a5_cross_protocol_effects.csv` | `results/summaries/protocol_{2,3}/` |
| What does the frozen representation show? | Figure 4 | `results/embeddings/` metrics, coordinates, and manifests |
| Where does a paper number come from? | `paper/evidence_manifest.csv` | Its `source_path` and `selector` columns |

## Protocol roles

| Protocol | Role | Epochs | Batch size | Peak learning rate |
|---|---|---:|---:|---:|
| 1 | Canonical | 50 | 16 | 1 × 10⁻⁴ |
| 2 | Robustness | 300 | 64 | 1 × 10⁻³ |
| 3 | Robustness | 100 | 32 | 5 × 10⁻⁵ |

Protocol 1 defines the primary adaptation curve. Protocols 2 and 3 test whether its direction changes under substantially different optimization budgets. Do not pool their runs into one estimate.

## Per-run directory anatomy

A typical run is:

```text
results/protocol_1/oxide/N50_seed0/finetune_last2/
├── config.json
├── history_train.json
├── history_val.json
├── ids_train_val_test.json
├── prediction_results_test_set.csv
├── prediction_results_train_set.csv
└── summary.json
```

The public release intentionally omits checkpoints, duplicated structure files, and run-console logs. The fixed configs, identifiers, histories, summaries, and predictions are sufficient to inspect the result; checkpoints are regenerated by training.

The custom fine-tuning and scratch trainers instantiate `torch.nn.L1Loss` directly. Their inherited upstream-style JSON configs retain a `criterion` field for schema compatibility, but that field is not read by these custom entry points.

### `summary.json`

Important fields:

- `n_train`, `n_val`, `n_test`: exact split sizes used by the run;
- `epochs`, `batch_size`, `learning_rate`: optimization settings;
- `unfrozen_groups` and `trainable_parameter_names`: the exposed final GCN block and output head;
- `best_epoch`: validation-selected checkpoint epoch; epoch 1 means the end of the first epoch, not the untouched zero-shot model;
- `test_mae_eV_per_atom`: primary fixed-test result;
- `elapsed_seconds`: hardware-dependent runtime;
- `batchnorm_invariant_report`: evidence that frozen batch-normalization state did not drift.

### Histories

`history_train.json` and `history_val.json` store epoch-order loss/metric arrays. The selected epoch is the unique validation-loss minimum used by the aggregation scripts. A late best epoch indicates continued validation engagement; an epoch-1 selection indicates that later updates were not preferred by validation.

### Predictions

`prediction_results_test_set.csv` is the run-level numerical authority for test MAE and structure-level comparisons. Rows join to zero-shot predictions by JARVIS identifier. `prediction_results_train_set.csv` supports diagnostics but is not used as fixed-test evidence.

## Metric direction

The repository uses two related differences with opposite-looking names:

| Metric | Definition | Positive means |
|---|---|---|
| `delta_adapt` | fine-tuned MAE − zero-shot MAE | Fine-tuning is worse |
| `transfer_gain_vs_zero_shot` | zero-shot MAE − fine-tuned MAE | Fine-tuning improved |

Therefore, a negative `transfer_gain_vs_zero_shot` agrees with a positive `delta_adapt`: both say fine-tuning increased error. MAE values are in eV/atom. Seed variation is reported as the sample standard deviation (`ddof = 1`) unless a table explicitly labels another interval.

## Summary layers

`results/summaries/protocol_<n>/finetune/` contains one row per run plus family-by-N aggregates. The top-level A1–A6 tables answer cross-cutting questions:

- A1: adaptation effect sizes and seed consistency;
- A2: zero-shot residual decomposition;
- A3: structure-level heterogeneity;
- A4: learning dynamics and checkpoint timing;
- A5: cross-protocol effect magnitudes;
- A6: promoted representation and distance–error robustness results.

The supplementary `paper/supplementary/data/all_240_training_runs.csv` is the compact inventory across 180 fine-tuning and 60 scratch runs. Only the 20 Protocol 1 scratch runs at N = 50 and 500 are part of the main-scope pretrained-versus-random comparison; the other 40 are retained robustness evidence.

## Zero-shot authorities

The oxide and nitride prediction CSVs contain target, prediction, and absolute-error values for the fixed tests. The headline values are:

| Family | Test structures | MAE (eV/atom) |
|---|---:|---:|
| Oxide | 1,484 | 0.034184 |
| Nitride | 242 | 0.069542 |

The nitride/oxide ratio is 2.03 with a 95% structure-bootstrap interval of [1.70, 2.42]. This is an observed family performance gap, not a causal attribution or a formal distribution label.

## Embedding artifacts

`results/embeddings/embeddings/` stores frozen arrays, while `results/embeddings/manifests/` defines row identity and extraction settings. Public metric authorities are `paper/supplementary/data/embedding_layer_metrics_overall.csv`, `paper/supplementary/data/pure_oxide_embedding_sensitivity_deltas.csv`, and `paper/figures/figure4_distance_error_statistics.csv`; the projection figures are descriptive. Statistical claims use raw-space metrics and prespecified distance analyses rather than visual separation in a two-dimensional projection.

The primary within-nitride result is a positive Spearman association between absolute error and five-nearest-oxide distance (`rho = 0.346`, 95% CI [0.225, 0.463]). It is correlational and protocol-specific.

## Evidence manifest

Each row of `paper/evidence_manifest.csv` records an identifier, description, displayed value, units, public `source_path`, and selector. The path is the authority; the selector identifies the row, field, or aggregation. Use it when checking manuscript prose, tables, captions, or supplementary claims.
